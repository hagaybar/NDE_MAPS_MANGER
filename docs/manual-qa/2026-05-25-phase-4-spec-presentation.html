Reconcile Wizard Redesign

Make reconcile shelf-centric and non-destructive by default — so a librarian can rename, add, and remove shelves through a full edit cycle without losing data or needing an engineer.

The core idea

1 · Removing a shelf stops destroying its data

Today, removing a shelf in reconcile deletes its library row forever — collection ranges, descriptions, all gone. The fix: a soft-delete — the row is set aside in a recoverable “unlinked” pool, reassignable later.

✗ Today

remove shelf CC_1 → CSV row (ranges, descriptions) gone

✓ Phase 4

remove shelf CC_1 → 📦 row set aside in "unlinked" pool → reassign to CC_9 later
Why this is the heart of itA "remove" becomes safe and reversible. Everything in spec 4c hangs off this single idea.

Recovery

3 · Getting a set-aside row back

The CSV editor gains an “Unlinked entries (N)” panel. Each row has a “Reassign to shelf …” action that validates the target shelf id exists on that floor before saving — so a recovered row can never re-introduce a broken reference.

GuardrailReassign (and the inline "add entry") both enforce the bundle invariant: the svgCode must resolve to a real shelf on its floor, or the save is rejected with a clear message.
The new wizard

4 · Grouped by what changed — nothing destructive by default

The wizard reads top-to-bottom, grouped by change type. Each item has safe, explicit actions:

↺ Renamed via Phase 3

  • Confirm — apply old → new to the CSV
  • Treat as separate add + remove
  • Ambiguous → pick from candidate previews

＋ Added #57

  • Add library entry now (inline form)
  • Leave unmapped (decorative/later)
  • Remove from the map (undo before promote)

－ Removed #59

  • Unlink (keep data) → the pool
  • Cancel removal → revert that floor's SVG
The mental-model fixA librarian can answer, in plain language: “what changed, and what do I want to do about each?” — which today's wizard can't.
How we'll build it

5 · Three shippable stages

4a · ships first
Renames
Confirmable rename pairs (uses Phase 3 + the existing backend rename action). Smallest, highest immediate value. (rename half of #59)
4b · #57
Adds
The three new-shelf actions: add inline / leave / remove-from-SVG. Needs an add-entry form + a staged-SVG edit.
4c · #59
Soft-unlink pool
The recoverable pool + CSV reassign panel + 90-day lifecycle cleanup. The largest; the heart of #59.
OrderSmallest-first by value & risk. Each is its own spec → plan → PR, tracked by a meta-issue. 4a/4b don't need the pool; 4c is self-contained.
Two things still open (resolved in each sub-plan)(a) the exact "remove from SVG" edit mechanism (4b); (b) whether pool files keep all CSV columns (default: yes, lossless).
