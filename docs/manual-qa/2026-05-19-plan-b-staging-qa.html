Reconcile path: upload SVG that removes a referenced shelf
Critical ~5-7 minutes
0

The whole point of the staging flow is to catch CSV ↔ SVG drift before it goes live. This test exercises the reconcile wizard, which is the user's escape hatch.

Steps
  1. Pick a real svgCode currently referenced by the CSV (CSV Editor → copy one from a row on floor 1, e.g. CC_X).
  2. Edit the local floor_1.svg from the previous test — remove or rename the <rect id="CC_X"> element. Save.
  3. In the admin SPA, Replace floor_1.svg with this broken version.
  4. Validate should fail. The reconcile wizard should open showing CC_X as a removed ref.
  5. Choose "Rename" → pick a still-existing shelf (e.g. CC_Y), OR "Delete" → confirm.
  6. Apply reconcile. Validate should now pass.
  7. Promote.
  8. Open CSV Editor, find the row that was originally CC_X — confirm the rename took (or the row is gone if you chose delete).
Expected: Wizard surfaces the conflict clearly. Reconcile mutates the staging CSV (not prod). Re-validation passes. Promote copies the new SVG + the patched CSV to prod atomically.

Lock contention: second admin blocked by 423
Important Needs 2 sessions · ~3 minutes
0

Only one admin can stage at a time. Without this guarantee, two concurrent uploads would race and corrupt staging state.

Steps
  1. In Session A (your normal admin), start a Replace. Don't promote — leave it staged.
  2. Open a private/incognito window. Sign in as a different admin user (or the same user in a different browser profile so the lock owner identity differs).
  3. In Session B, navigate to SVG Manager. Try to Replace any file.
  4. Capture B's error response (network tab → upload call) and the UI message.
  5. Back in Session A: Promote OR Discard to release the lock.
  6. Retry B's upload — should now succeed.
Expected: Session B's upload returns HTTP 423. UI shows "staging owned by <sessionA-user>" or similar clear message. After A releases, B can proceed.
