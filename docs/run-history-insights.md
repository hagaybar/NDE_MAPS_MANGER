### Key Decisions & Insights

- **Strict plan adherence is the dominant pattern.** Every feature/fix run cited an
  explicit task-by-task plan under `docs/superpowers/plans/` and executed it task-by-task
  (t0…tN). Deviations were rare, small, and explicitly logged (e.g. SoT t3 dropped a
  `@jest-environment node` directive because admin setup needs jsdom; t10/t12 had
  "minor plan deviation" notes). The plan doc was updated when reality diverged.
- **Shared-module parity with fixtures is how CSV↔SVG drift is prevented.** The SoT
  foundation deliberately built **two implementations** of each rule — server
  (`lambda/shared/svg-shelves.mjs`, `lambda/shared/validateBundle.mjs`) and client
  (`admin/services/svg-shelves.js`, `admin/services/bundle-validator.js`) — guarded by
  parity tests over shared fixtures (`lambda/__tests__/fixtures/{svg-shelves,bundles}/`).
  This is now encoded in CLAUDE.md as a hard rule.
- **The `?v=N` cache-bust suffix was actively harmful and was removed (#24), then
  enforced as forbidden (#9 regression guard).** Bare URL imports are now mandatory;
  SoT t10 explicitly notes the issue-#24 regression guard "forbids `?v=5`". This is the
  module-duplication root cause — duplicate module instances from versioned imports.
- **map-object typing (#16 PR1) renamed the domain vocabulary** from `shelf*` to
  `location*`/`map-object` across `map-editor.js` while preserving cross-module prop
  keys and callback param names — a careful rename that kept the Jest signature
  identical to main.
- **TDD red→green was followed literally.** Almost every run committed failing tests
  first (t1 "red phase"), then implementation (t2+). The red commit was a verifiable
  artifact (module-not-found / TypeError on `__test`).
- **Pre-feature tag + feature branch + PR-no-merge is the standard envelope.** Every
  fix run created a `pre-issue-NN` tag on main and a `fix/feat-issue-NN-*` branch, pushed,
  opened a PR, and explicitly did **not** merge. SoT additionally used a named pre-feature
  tag (`pre-sot-foundation`) and did **not** push at all per instruction.

### What Worked

- **Stable test signature as a regression oracle.** Every run measured success against
  "the same N pre-existing issue-#9 failures, no new failures" (14 Jest failures across 4
  suites: validation, data-model, user-menu, edit-user-dialog). This gave a crisp,
  repeatable green/red gate without needing the legacy failures fixed first.
- **Isolated spot re-runs of new + anchor specs** after the slow full Playwright suite.
  Several runs (#15, #35) ran the full suite to exit 0 under heavy contention (40–60 min),
  then re-ran just the new spec + a few anchor specs in isolation (seconds) to confirm
  green without contention noise.
- **en/he i18n parity in lockstep.** Every UI run updated `en.json` and `he.json`
  together and validated both; #23 specifically proved the RTL (he-admin) closed-panel
  invariant with a dedicated spec across both projects.
- **Self-correcting agents within a task.** #35 t2 caught a missing `toast-container`
  in test setup and fixed it by amending the prior red commit with a polyfill + container
  div, rather than failing the task. #14-phase-2a t7 guarded `scrollIntoView` with a
  `typeof` check after a test surfaced its absence.
- **Multi-PR decomposition of a large issue.** Issue #16 was split into PR1 (data-typing
  foundation) and PR2 (UX on top of the foundation), each its own run with its own
  pre-feature tag. PR2 explicitly verified "PR 1 foundation" before starting.

### What Didn't Work / Friction

- **The Jest 30 vs 29 trap.** #16-PR2 t3 hit a hard wall: `npx jest` pulled an
  incompatible Jest 30 and the red tests behaved wrong. Fix was to invoke the project's
  pinned Jest 29 directly: `node admin/node_modules/jest/bin/jest.js`. Worth pinning the
  invocation in process scaffolds so this doesn't recur.
- **Stale `?v=` mock suffixes leaked into tests.** #16-PR2 t4 had to "retro-fix" the
  Task 3 test file to drop `?v=` mock suffixes — the cache-bust pattern from #24 was still
  echoed in test mocks even after the source imports were cleaned.
- **Floor SVG migration left stale ids.** #16-PR1 t2's regex migration of `floor_{0,1,2}.svg`
  marked 1/202/191 elements but warned on 3 stale ids, requiring a `KNOWN_STALE_ROWS` skip
  list in the alignment test — i.e. the CSV and SVGs were not perfectly in sync and the
  test had to tolerate known drift.
- **Full Playwright suite is slow and contention-sensitive** (56.9 min in #35, "slow under
  contention" in #15). It runs to exit 0 but isn't a fast feedback loop; the isolated
  spot re-run is the real gate.
- **Snapshot churn from CSS additions.** #14-phase-2a t10 had to rebaseline 3 phase-3 UX
  snapshots that shifted purely because the orphan-panel CSS was appended — unrelated
  visual baselines are fragile to additive CSS.
- *(Out of scope but noted per request:)* A later run `01KSF754 fix-50-map-editor-refresh`
  **FAILED** on a separate day; the #50 redo was abandoned and the fix re-approached via
  the poll-until-fresh path (PR #66, per MEMORY). Not among these 10.

### Recommendations

1. **Pin the Jest invocation in process scaffolds** to `node admin/node_modules/jest/bin/jest.js`
   (or an npm script) so runs never accidentally pick up a globally-resolved Jest 30. This bit
   #16-PR2 mid-run.
2. **Add a repo-wide grep guard against `?v=` in imports AND test mocks**, not just source.
   #24 cleaned source imports but #16-PR2 still found stale `?v=` in test mocks. The
   regression guard should cover `__tests__/` too.
3. **Keep building rules as shared server+client modules with parity fixtures.** The SoT
   foundation's twin-implementation + parity-test approach is the right antidote to CSV↔SVG
   drift; extend it to any future rule that must run in both Lambda and the browser.
4. **Treat "same pre-existing issue-#9 failures, no new" as the explicit success contract**
   for every run until #9 is actually fixed — but schedule #9 (the 14 failing tests across
   validation/data-model/user-menu/edit-user-dialog) because it's now masking the true
   baseline in every single run's verification step.
5. **Isolate additive CSS from visual baselines** or expect to rebaseline. Consider scoping
   new-component CSS so it can't reflow existing snapshotted layouts, or accept a
   rebaseline step in the plan for any CSS-touching run.
6. **Continue the pre-feature-tag + branch + PR-no-merge envelope** — it held cleanly across
   all 8 feature/fix runs and gives the human a reviewable, rollback-safe artifact every time.

### Recommendations

1. **Guard process entrypoints against missing inputs.** Both RUN_FAILED cases were `TypeError` at the top of `process()` on properties of `undefined`. Add `inputs ?? {}` defaults and validate required keys early so failures land in a task with a useful message instead of crashing the orchestrator.
2. **Stop the `-v2` pattern; fix in place.** Renaming `admin-client-logging` to `admin-client-logging-v2` leaves orphaned process files and split run history. Keep the same processId, fix the bug, re-invoke.
3. **Standardize on plan + pre-feature tag + branch for any run with >5 tasks.** This is already informal practice (`map-editor`, `map-editor-ux-polish`) and matches MEMORY.md. Make it explicit in process scaffolds.
4. **Backfill coverage on legacy Lambdas** to unblock the 95% global target — `deleteSvg.mjs`, `listSvg.mjs`, `putCsv.mjs`, `uploadSvg.mjs` were called out as untested.
5. **Capture visual baselines for en + he × admin + editor as a 2×2 matrix** in every UI run, the way `map-editor-ux-polish` did. It's the cleanest gate against RTL regressions.
